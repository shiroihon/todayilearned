What

API

「システムやサービス同士がデータをやり取りするための『約束事（インターフェース）』」
例えるなら、
「注文（フロントエンド）と厨房（バックエンド）をつなぐ『伝票』のようなもの」
つまりは、
「ソフトウェアの機能を外部から安全に利用できるようにする仕組み。HTTPリクエストを通じてデータを要求・操作する方法を定義したもの」

sequenceDiagram
  フロントエンド->>+バックエンド: HTTPリクエスト（注文データ）
  バックエンド-->>-フロントエンド: 即時レスポンス（成功/失敗）

認識齟齬の可能性があるポイント

❌ 誤解されがちな考え方

  • 「APIはバックエンド開発者が作るもの」
    • → 実際はフロントエンドや外部連携も含む全体的な設計が必要
  • 「APIはコーディング段階（開発フェーズ）だけの話」
    • → 設計やテスト、運用まで含む継続的なプロセス ✅ 正しい理解
  • APIは「開発プロセスの横断的な要素」であり、各フェーズで異なる役割
  • 特に現代開発では、API-Firstアプローチ（設計を最優先）が主流

APIは特定のフェーズに属さず、開発ライフサイクル全体に関与する。 特に重要なのは設計段階で、ここで明確な契約（仕様）を定義しないと後続工程で問題が発生。 現代開発では「API-First」（仕様を最初に作る）がトレンド。 認識齟齬を防ぐためには、APIを「プロダクト全体のインターフェース設計」と捉え、チーム全体で仕様を共有することが不可欠。

MCP

APIは「何を（データ/機能）提供するか」の契約書 MCPサーバは「どのように（モデルを）実行するか」のインフラ 例えるなら、 APIが「宅配注文のメニュー」で、MCPサーバが「キッチン設備」 両者は役割が異なるものの、組み合わせることで高度なサービスを構築できる

API vs MCPサーバ 比較表

\ API MCPサーバ
アナロジー 「飲食店のメニュー」（注文方法を標準化） 「シェフのキッチン」（料理を実際に調理する環境）
定義 システム間のデータ連携を実現する「インターフェース（契約）」 計算モデル（AI/シミュレーション）を実行・管理する「専用サーバ環境」
目的 外部システムに対して機能やデータを提供 複雑なモデル（例: 気象予測、金融リスク分析）を効率的に実行
プロトコル HTTP (REST/gRPC)、WebSocket、MQTT など 専用プロトコル（例: MCP独自規約、高性能計算向け通信）
データ形式 JSON、XML、Protocol Buffers など モデル入力/出力データ（数値データ、バイナリ形式など）
利用例 - Twitter API（ツイート取得）
- Stripe API（決済処理）		 - 気象予測シミュレーション
- 金融リスクモデル計算
スコープ 汎用的（あらゆるシステム間連携に使用可能） 特化的（特定の計算モデル実行に最適化）
リソース管理 通常、リクエストごとに独立した処理（ステートレス） 計算リソース（CPU/GPU/メモリ）を集中的に管理
拡張性 エンドポイント追加で機能拡張可能 サーバクラスタや分散計算でスケールアウト
代表ツール Postman、Swagger、FastAPI 専用モデル実行フレームワーク（例: TensorFlow Serving、HPC環境）
